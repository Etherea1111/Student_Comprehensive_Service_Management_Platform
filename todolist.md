# 项目完成度待办清单

更新时间：2026-05-25  
审查依据：`README.md`、`Detailed_Requirements_Specification.md`、当前后端服务、数据库脚本、小程序页面与开发文档。  
范围说明：模块五“学业情况分析与预警”此前已被甲方认定不需要实现，因此本文不把它列为业务开发待办，但会把 README/数据库设计文档中仍保留模块五口径的问题列为文档一致性待办。

## P0：影响核心验收或真实落地

| 序号 | 功能 | 当前完成度不高的原因 | 下一步优化方向 |
| --- | --- | --- | --- |
| P0-01 | PostgreSQL 数据库实际接入与迁移验证 | 当前新增了多批表和字段，已确认演示服务器使用 PostgreSQL；很多功能只有接口和脚本，需要继续在服务器数据库联调。 | 在 PostgreSQL 环境执行 `backend/db/schema.sql`，整理一次可重复迁移流程，补充初始化检查脚本、权限授权说明和迁移失败回滚说明。 |
| P0-02 | 微信实名登录与账号绑定 | README 要求基于微信账号实名制；当前主要是账号密码注册、登录和学号姓名绑定，未接入 `wx.login`、openid/session_key。 | 增加小程序 `wx.login` 到后端 code2Session 流程，将 openid 与 `users` 一对一绑定，再保留账号密码作为调试或备用入口。 |
| P0-03 | 模块一：知识库文件解析入库 | 现在支持文件上传和知识条目附件关联，但 Word/PDF/Excel 文件不会解析为知识条目，也没有“初始打包文件库”导入流程。 | 建立文件导入任务：上传文件、抽取文本、生成待复核知识草稿、保留原文件和来源；优先支持 docx/xlsx/pdf 的文本抽取。 |
| P0-04 | 模块一：智能问答语义匹配 | 当前搜索是分词后的 SQL `ILIKE` 加权排序，不是 README 所说的 AI/语义匹配；长问题、同义表达和上下文追问仍较弱。 | 在“标准答案优先”的前提下增加全文索引或向量检索；所有答案仍必须绑定知识条目和官方来源，避免直接自由生成。 |
| P0-05 | 模块四：证明文件模板填充 | 当前证明生成主要是文本内容和轻量 PDF 输出，不是按学校/学院标准 Word/PDF 模板填充字段。 | 增加证明模板管理，按模板字段映射学生信息，生成可预览 PDF 和可下载 docx/pdf，并保留生成版本。 |
| P0-06 | 模块四：分级审批角色边界 | 已有两级状态流转，但权限只看 `approve_requests`，未严格区分“辅导员审批”和“学院复核”角色；学院领导默认权限里也未自然获得审批能力。 | 为审批节点配置允许角色/权限，例如 counselor 只能提交学院复核，college_leader 或 super_admin 才能最终通过。 |
| P0-07 | 模块二：关键节点自动提醒 | 当前有到期提醒查询列表，但没有定时任务、通知队列写入或学生端自动提醒送达。 | 增加每日定时扫描 `process_progress.next_deadline`，生成提醒记录，并复用公告/通知投递通道推送给学生和负责人。 |
| P0-08 | 敏感字段加密覆盖 | 身份证、手机号、生源地、户籍已加密，但导师、学籍状态、休学/延毕、备注等仍主要是明文字段加权限遮罩；Detailed 明确这些属于保密信息。 | 梳理敏感字段等级，对导师、特殊学籍状态、延毕/休学说明等增加加密或拆分敏感扩展表，并统一审计访问。 |
| P0-09 | 后台管理端形态 | README 要求管理员端便于 PC 统计分析、导入导出；当前管理能力主要放在微信小程序管理员页，PC 端后台未实现。 | 建立 PC Web 管理端，优先覆盖学生信息、知识库、模板、流程、公告、审批、导入导出、操作日志和统计看板。 |

## P1：重要功能已有基础，但仍偏粗糙

| 序号 | 功能 | 当前完成度不高的原因 | 下一步优化方向 |
| --- | --- | --- | --- |
| P1-01 | 模块三：邮件真实发送 | 已有 `email` channel 和投递记录，但 provider 仍返回 `pending`，没有 SMTP/API 配置、失败重试和发送状态回写。 | 接入学院 SMTP 或邮件服务 API，增加环境变量、模板渲染、失败重试、退信状态和投递日志。 |
| P1-02 | 模块三：微信订阅消息 | 已有 `wechat` channel 预留，但没有订阅授权、模板 ID、access token 获取、发送接口和回调处理。 | 接入微信订阅消息流程，保存用户订阅状态，配置模板 ID，并将发送结果写回 `announcement_deliveries`。 |
| P1-03 | 模块三：官方来源自动同步 | 目前官方来源只能维护链接并生成待复核草稿，不会抓取官网/公众号正文，也没有去重策略以外的内容聚合。 | 优先支持 RSS/官方网页白名单解析；生成草稿时提取标题、正文、发布时间和附件，并要求管理员复核发布。 |
| P1-04 | 模块三：精准推送配置体验 | 目标人群支持多种 `type:value`，但仍依赖手输文本，容易输错；没有人群预估、条件组合预览和保存常用人群。 | 增加可视化目标选择器、目标人数预估、条件校验、常用人群模板和发布前确认。 |
| P1-05 | 模块一：知识反馈闭环 | 已能提交反馈和标记处理，但“处理”只是状态变化，没有自动转为知识草稿、关联修订版本或通知反馈人。 | 将反馈转为待办工单，支持关联知识条目、创建修订草稿、处理意见和反馈结果通知。 |
| P1-06 | 模块一：敏感知识展示策略 | 当前知识条目只有 `sensitive_hint` 文本提示，未严格做到敏感内容只给官方链接或按角色隐藏正文。 | 增加 `is_sensitive`/可见角色字段，搜索接口根据用户权限返回正文、摘要或仅返回官方链接。 |
| P1-07 | 模板文件管理 | 模板上传复用了知识库文件上传，模板表只保存 URL/元数据；缺少独立文件权限、下载统计、版本管理和签名链接。 | 拆出模板文件存储表，支持模板版本、下载审计、替换文件、失效归档和受控下载链接。 |
| P1-08 | 导入导出格式覆盖 | 当前导入主要覆盖学生、流程、题库，导出主要是 CSV；README 要求 Excel/Word/PDF 等导入导出。 | 增加 xlsx 导出、审批/证明 PDF 批量导出、知识库/模板批量导入，并提供下载模板和字段校验报告。 |
| P1-09 | 学生信息维护闭环 | 已有学生导入和列表展示，但缺少管理员端单个学生编辑、特殊状态变更记录、毕业/校友动态维护流程。 | 增加学生档案编辑页、状态变更日志、批量毕业转校友、休学/复学/退学流转和敏感状态审计。 |
| P1-10 | 操作日志完整度 | 已有通用操作日志和部分敏感访问记录，但多数业务没有记录变更前后值，也缺少日志筛选、导出和保留策略。 | 在关键更新接口补充 before/after 快照，增加日志查询过滤、导出、保留期归档和异常操作告警。 |
| P1-11 | 模块二：党团日常工作记录 | 已有工作记录台账，但偏文本录入；缺少附件材料、参与人校验、统计分析和与党团流程节点关联。 | 支持材料附件、参与学号校验、活动类型统计、学生参与记录沉淀，并与流程进度更新关联。 |
| P1-12 | 模块二：流程进度历史 | 当前可维护当前阶段和已完成动作，但缺少完整阶段变更历史、每次活动批量变更留痕和学生端历史时间线。 | 增加 `process_progress_logs`，记录前后阶段、操作人、原因、导入批次和相关活动记录。 |
| P1-13 | 模块四：审批保留期和归档 | Detailed 提到审批记录大约保存一学期；当前没有自动归档、清理策略或导出归档包。 | 增加审批归档状态、按学期导出、附件归档、超期提醒和清理任务。 |
| P1-14 | 模块四：盖章附件在线预览 | 当前审批老师可下载附件，但没有在小程序内按类型预览、附件审计、涉密附件替代说明的明确审批提示。 | 接入 `wx.openDocument`/预览接口，记录附件下载日志，对涉密说明与附件缺失给出审批端提示。 |
| P1-15 | 账号权限维护细粒度 | 已能维护账号、角色和额外权限，但权限项没有可视化选择器，缺少角色模板、操作二次确认和权限变更日志。 | 增加权限字典、角色模板、权限勾选 UI、账号变更 before/after 日志和禁用账号确认流程。 |

## P2：可用性、质量和文档一致性问题

| 序号 | 功能 | 当前完成度不高的原因 | 下一步优化方向 |
| --- | --- | --- | --- |
| P2-01 | 小程序离线 mock 兜底 | 多个 service 在 API 失败后静默回退到 mock 或本地缓存；真实联调时可能掩盖后端错误。 | 区分“开发离线模式”和“生产 API 模式”，生产模式下请求失败应明确报错，不自动混用 mock 数据。 |
| P2-02 | 理论自测管理 | 理论自测题库可导入和作答，但管理端题目审核、分类维护、错题复习、成绩统计较弱；README 中虽为可选，但当前展示为已实现模块。 | 增加题目管理页、发布/下架、错题本、答题记录统计和按班级/年级汇总。 |
| P2-03 | 标签体系 | 公告标签有表，知识库标签仍是文本字段；缺少统一标签字典、标签合并和停用机制。 | 建立统一标签管理，支持知识库、公告、模板共用标签，增加标签合并、别名和启停。 |
| P2-04 | 文件存储可靠性 | 上传文件保存在本地 `uploads`，缺少备份、清理、容量限制、病毒扫描和文件哈希去重。 | 增加文件存储配置层，可切换对象存储或共享目录；补充哈希、权限、清理和备份策略。 |
| P2-05 | 统计分析看板 | 管理员首页仅有少量数量指标，README 要求 PC 统计分析；目前没有趋势、分布、处理时效等统计。 | 增加知识反馈、公告触达、审批时效、党团进度、工作记录等统计图表和导出。 |
| P2-06 | API 文档 | `docs/backend-deployment.md` 只列了部分旧接口，未覆盖 P0/P1/P2 新增的导出、反馈、文件、账号、工作记录等接口。 | 生成完整 API 文档，列出路径、权限、请求体、响应体、错误码和示例。 |
| P2-07 | 数据库设计说明书 | `数据库设计说明书.md` 与当前 schema 明显不一致，还包含模块五表设计和旧表名。 | 按当前 `backend/db/schema.sql` 重写数据库说明书，移除或标注模块五为不实施范围。 |
| P2-08 | 用户手册与演示材料 | README 要求学生/管理员用户手册和演示 PPT；当前未见完整用户手册和 PPT。 | 补充学生端、管理员端操作手册，制作演示脚本和答辩 PPT。 |
| P2-09 | 自动化测试 | 当前主要依赖 `node --check` 静态语法检查，缺少服务层、路由、权限、导入导出和审批流程测试。 | 增加 Jest 或 node:test，覆盖核心 service；准备测试数据库和关键端到端用例。 |
| P2-10 | 前端表单校验 | 多数小程序表单只做必填校验，日期、目标人群、权限、学号列表等复杂字段校验较弱。 | 增加统一表单校验工具、字段级错误展示和提交前预览。 |
| P2-11 | 错误提示国际化/中文化 | 后端错误多为英文，如 `not found`、`Permission required`；小程序部分直接展示英文错误。 | 建立错误码和中文错误映射，前端按错误码展示可理解提示。 |
| P2-12 | 模块五文档口径 | README 和数据库设计说明书仍写有模块五，开发文档又说明模块五不进入主流程，容易造成验收范围歧义。 | 在 README 中明确模块五已由甲方排除，或移入“后续扩展/不交付范围”。 |

## 不建议继续投入的事项

| 功能 | 原因 | 建议处理 |
| --- | --- | --- |
| 模块五学业分析与预警 | 详细需求中明确“不推荐加入任课老师评价、选课推荐等”，且此前甲方已认定模块五不需要实现。 | 不继续开发业务功能，只做文档范围说明，避免验收时被误认为缺失。 |
| 短信通知 | Detailed 表示微信和邮件足够，短信不一定必须做；真实短信还涉及服务商、费用和手机号授权。 | 暂不实现短信，把通知重点放在小程序站内消息、微信订阅消息和邮件。 |
| 自由生成式 AI 问答 | README 强调避免大模型幻觉，当前业务更看重标准答案和官方链接。 | 只做检索增强和老师复核，不开放无来源自由回答。 |

## 建议实施顺序

1. 先完成真实 PostgreSQL 接入、迁移验证、微信登录绑定和核心数据联调。
2. 再补模块一文件解析入库、语义检索和模板文件版本管理。
3. 随后补模块二自动提醒、流程历史和工作记录附件。
4. 接着补模块三邮件/微信真实投递、官方来源解析和精准推送配置器。
5. 最后补模块四标准模板生成、审批角色边界、归档策略，以及 PC 管理端和交付文档。
